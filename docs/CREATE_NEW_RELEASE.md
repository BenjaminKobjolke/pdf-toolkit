# Creating a New Release

How to cut a new pdf-toolkit release: bump the version/build, write release
notes, and build the distributable. The in-app **Release Notes** view reads the
notes that ship inside the build.

---

## 1. Version & build number

A release is identified by the label **`<version>_<build>`**, e.g. `0.1.0_22`.

| Part | Source of truth | How to read |
|------|-----------------|-------------|
| `version` (semver) | `pyproject.toml` ŌåÆ `[project] version` | `tools\version_get.bat` prints the full `<version>_<build>` label |
| `build` (integer) | `build_version.txt` (project root) | `tools\build_get.bat` |

**Model ŌĆö bump first, ship next.** `build_version.txt` holds the **last shipped**
build (`0` = nothing shipped yet). Cutting a release increments it and ships that
new number, so the release ships as `<version>_<build_get + 1>`. Author the notes
folder for that next label.

**Get the current label / numbers:**

```bat
tools\version_get.bat   :: prints e.g. 0.1.0_22
tools\build_get.bat     :: prints e.g. 22
```

**Increment / decrement the build number:**

```bat
tools\build_increment.bat   :: 22 -> 23 (do this before each new release)
tools\build_decrement.bat   :: 23 -> 22 (undo)
```

**Bump the semver version:** edit `version = "..."` in `pyproject.toml` by hand
(major/minor/patch is a human decision). Reset/keep the build number as you see
fit ŌĆö the label is always `version_build`.

> Under the hood the bats call `python -m app.release.build_number get|increment|decrement`
> and `python -m app.release.release_info`. The logic lives in
> `app/release/build_number.py` and `app/release/release_info.py`.

## 2. Create the release-notes directory + `en.json`

One folder per release under `release_notes/`, named with the label:

```
release_notes/
  0.1.0_22/
    en.json        <-- author THIS only
    de.json        <-- generated by the translation tool (step 3)
```

Create `release_notes/<version>_<build>/en.json` with this schema. The actual
release text goes in the **`notes`** array:

```json
{
  "version": "0.1.0",
  "build": 22,
  "date": "2026-06-19",
  "title": "Short headline for this release",
  "notes": [
    "First user-facing change",
    "Second user-facing change"
  ]
}
```

| Key | Meaning |
|-----|---------|
| `version` | semver, matches the folder + `pyproject.toml` |
| `build` | integer, matches the folder + `build_version.txt` |
| `date` | release date, `YYYY-MM-DD` |
| `title` | one-line headline shown in the dialog header |
| **`notes`** | **array of strings ŌĆö the actual release notes** (one bullet each) |

**Author only `en.json`.** Do not hand-write other languages.

The JSON keys are defined once in `app/release/schema.py` ŌĆö do not hardcode them
elsewhere.

---

## 3. Translate (DO NOT SKIP) ŌÜĀ’ĖÅ

After writing `en.json`, generate the other languages by running:

```bat
tools\translator_release_notes.bat
```

This runs the shared `GPT-json-translator` tool with `--translate-recursive=en.json`,
so it walks **every** `release_notes/<label>/en.json` and writes the other locales
(`de.json`, ŌĆ”) next to each one, using the same schema with `title`/`notes`
translated. **This step is mandatory** ŌĆö
skipping it means non-English users see English (the loader falls back to
`en.json`, so nothing breaks, but translations will be missing).

---

## 4. Build the release

Release notes are **bundled inside the exe** (via `FastFileViewer.spec` `datas`), so the
in-app view works from a standalone build ŌĆö they are not merely copied next to it.

```bat
tools\build_increment.bat   :: 1. bump the build number (if not already done)
tools\build_release.bat     :: 2. test + build + stage the exe
```

`tools\build_release.bat`:
1. runs the unit tests (aborts on failure),
2. builds `dist\FastFileViewer\` via `tools\build_exe.bat` (PyInstaller, onedir,
   windowed) ŌĆö this is where `release_notes/` + `build_version.txt` get bundled,
3. **signs `dist\FastFileViewer\FastFileViewer.exe` in place** via
   `tools\presign_exe.py` (release-tool's network-share signer ŌĆö requires
   `//XIDA-SERVER/SigningExecutables/` reachable, config in
   `tools\publish_settings.ini` `[PreSigning]`),
4. compiles the **Windows installer** from `installer\installer.nsi` (NSIS,
   `makensis` must be on PATH) and renames it to
   `releases\windows\FastFileViewer_v<version>_<build>.exe` ŌĆö the versioned
   filename is the version stamp; the NSI itself carries no version info,
5. mirrors the folder to `release\<version>_<build>\FastFileViewer\`.

The installer packages the **GUI viewer only** (`dist\FastFileViewer`), not the
CLI tools. It installs to Program Files, adds Start Menu + uninstall shortcuts.

To build only the exe without signing/installer/staging, run
`tools\build_exe.bat` directly (dev builds stay sign-free).

---

## 4b. Publish

```bat
tools\publish_release.bat
```

Publishes the installer under the **stable name `FastFileViewer.exe`** (constant
download URL): the bat copies `releases\windows\FastFileViewer_v<label>.exe` to
`releases\windows\FastFileViewer.exe` and hands that to release-tool. The
`[PreSigning]` pipeline signs the installer (network-share handshake, signer CN
verified), then uploads it per `tools\publish_settings.ini`. Both the installer
**and** the FastFileViewer.exe inside it end up signed.

The previous remote `FastFileViewer.exe` is archived to
`versions/<--previous-version>/` before being replaced ŌĆö update the
`--previous-version` value in the bat to the last **shipped** label when cutting
a release. Run `tools\build_release.bat` first ŌĆö the bat aborts if the installer
for the current label doesn't exist.

The publish step also uploads every `release_notes/<label>/*.json` to the
website assets path (`[ReleaseNotes]` in `tools\publish_settings.ini`).

## 5. In-app Release Notes view

**Location:** **File ŌåÆ Release notesŌĆ”**, and the command palette (`Ctrl+Shift+P` ŌåÆ
"Show release notes / what's new"). The CLI wizard (`pdft`) also has a
"Show release notes" entry that prints them to the terminal.

**How it works** (`app/gui/release_notes_dialog.py`):
- Loads every `release_notes/<version>_<build>/<locale>.json` via
  `app/release/notes_loader.py`, sorted **newest first**.
- Shows the **latest release first**; **ŌåÉ Older / Newer ŌåÆ** buttons navigate
  through older releases. The body is a scrollable list of the `notes` bullets.
- Locale falls back to `en.json` when a translation is missing.
- Empty state ("No release notes yet.") when `release_notes/` has no entries.

## Quick checklist

1. `tools\build_increment.bat` ŌĆö bump build number.
2. (optional) edit `pyproject.toml` version for a semver bump.
3. `release_notes/<version>_<build>/en.json` ŌĆö write notes (`notes` array).
4. `tools\translator_release_notes.bat` ŌĆö generate translations. **ŌÜĀ’ĖÅ don't skip.**
5. `tools\build_release.bat` ŌĆö test, build, sign exe, build installer
   (`releases\windows\FastFileViewer_v<label>.exe`), stage
   `release\<label>\FastFileViewer\`.
6. Launch the exe ŌåÆ File ŌåÆ Release notesŌĆ” to verify the newest notes show.
7. `tools\publish_release.bat` ŌĆö sign + upload the installer as stable
   `FastFileViewer.exe` (plus the release-notes JSONs).
